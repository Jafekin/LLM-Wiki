# Research: 补充 Okapi BM25 原始研究与实现变体证据

# 补充 Okapi BM25 原始研究与实现变体证据

## 概述

**Okapi BM25** 是一种基于概率相关性框架的文档排序函数，用于估计文档与查询之间的相关性。它是现代 [[信息检索]] 系统中最常用的 [[词法检索]] 基线之一，并常作为 [[混合检索]]、[[重排序]] 及 [[RAG]] 检索阶段的候选召回方法。[3][5][13]

名称中的 “BM” 通常解释为 “Best Matching”。BM25 在 Okapi 系统及 TREC 评测背景下发展，随后被广泛纳入 [[Lucene]]/Elasticsearch 等搜索基础设施的默认或内置相似度实现。[3][5][13]

BM25 的核心目标是在传统 [[TF-IDF]] 的词项匹配基础上，同时处理：

1. 词项在文档中的重复出现，但使其贡献发生 [[词频饱和]]；
2. 文档长度差异，通过 [[文档长度归一化]] 抑制长文档因包含更多词而获得的不公平优势；
3. 稀有词通常更具区分能力，通过 [[IDF]] 加权；
4. 以较少、可调的参数获得实用的排序效果。[3][5]

有关 BM25 相比 [[TF-IDF]] 的定位和差异，参见 [[comparisons/bm25-与-tf-idf]]；基础算法说明参见 [[concepts/BM25]] 与 [[sources/BM25算法详解]]。

## 历史与原始研究脉络

### 概率相关性框架

BM25 源于 1970—1980 年代逐步发展的概率相关性框架。该框架与 Stephen E. Robertson、Karen Spärck Jones、S. Walker 等研究者密切相关，其思想是使用词项在相关与非相关文档中的出现特征来估计排序证据。[3][5]

Wikipedia 的概述将 BM25 的 [[IDF]] 成分追溯至 Binary Independence Model；即词项出现与否被视为与相关性有关的二元证据。[3] 但 BM25 并非严格从单一概率模型无近似地推导出的唯一公式，而是融合概率排序思想、经验近似与参数化词频/长度校正的实用排序函数。这一点也体现在 Robertson 与 Walker 的论文题目 *“Some Simple Effective Approximations to the 2-Poisson Model for Probabilistic Weighted Retrieval”* 中。[1][4]

### Okapi 与 TREC

Stanford 的课程材料指出，BM25 在 Okapi 系统语境中形成，并在 TREC 竞赛期间逐渐被其他团队采用。[5] 可追溯的早期文献包括：

- Robertson、Walker、Hancock-Beaulieu、Gull 与 Lau 的 *“Okapi at TREC”*，发表于 TREC-1（1992）。[1]
- Robertson 与 Walker 的 *“Some Simple Effective Approximations to the 2-Poisson Model for Probabilistic Weighted Retrieval”*，发表于 SIGIR 1994，页码 232–241。[1][4][5]
- Robertson、Walker 等人的 *“Okapi at TREC-3”*。[4]
- Beaulieu、Gatford、Huang、Robertson、Walker 与 Williams 的 *“Okapi at TREC-5”*，发表于 1997 年的 TREC-5，页码 143–165。[1]
- Spärck Jones、Walker 与 Robertson 的 *“A probabilistic model of information retrieval: Development and comparative experiments. Part 2”*，发表于 2000 年。[1][2][4][5]

因此，将 “Robertson et al. 1994” 作为常见 BM25 文献锚点是合理的；但若要完整理解其来源，应同时查阅 Okapi 的 TREC 系统论文、概率相关性框架研究以及 2-Poisson 近似研究，而不能把 BM25 简化为一篇论文中一次性完成的理论成果。[1][4][5]

## 标准评分形式

对于查询 $Q$、文档 $D$，常见的 BM25 文档评分形式为：

$$
\operatorname{score}(D,Q)
=
\sum_{q_i \in Q}
\operatorname{IDF}(q_i)
\cdot
\frac{f(q_i,D)(k_1+1)}
{f(q_i,D)+k_1\left(1-b+b\cdot\frac{|D|}{\operatorname{avgdl}}\right)}
$$

其中：

- $q_i$：查询中的第 $i$ 个词项；
- $f(q_i,D)$：词项 $q_i$ 在文档 $D$ 中的词频；
- $|D|$：文档长度；
- $\operatorname{avgdl}$：集合中的平均文档长度；
- $k_1$：控制词频非线性缩放，即 [[词频饱和]] 程度；
- $b$：控制 [[文档长度归一化]] 的强度；
- $\operatorname{IDF}(q_i)$：词项的逆文档频率权重。[3][5]

该形式构成通常所说的 BM25 核心；实际系统还可能增加查询词频权重、字段权重、短语或邻近度因素、相关反馈等扩展。[5]

### 常见 IDF 形式

[[Lucene]] 的 `BM25Similarity` 所使用的 IDF 形式为：

$$
\operatorname{IDF}(q_i)
=
\log\left(
1+
\frac{N-n(q_i)+0.5}
{n(q_i)+0.5}
\right)
$$

其中 $N$ 为参与统计的文档数，$n(q_i)$ 为含有该词项的文档数。[3][6][7][8]

这一形式保留了 `+0.5` 平滑校正，并额外在对数内部加入 `1+`。后者保证 IDF 为非负值，因而与某些文献或实现中不含 `1+` 的 Robertson/Sparck Jones 风格 IDF 公式并不完全相同。不同 IDF 公式在高频词、停用词及小型语料场景下可能造成不同的分值与排序行为。[3][4][6]

## 参数语义与极限行为

### `k1`：词频饱和参数

`k1` 控制词频的非线性归一化程度。[5][6][14]

- `k1 = 0` 时，词频贡献退化为二元出现信号：词项出现与否重要，重复次数不再增加分数。[5]
- `k1` 较大时，词频影响更接近原始词频，饱和更慢。[5]
- 常见经验范围是约 `1.2` 至 `2.0`。[3][5]

BM25 不直接采用线性词频的原因是：同一词项从出现 0 次变为 1 次通常具有较大检索意义，而从出现 20 次变为 21 次所新增的相关性证据往往较小。这种递减收益即 [[词频饱和]]。

### `b`：文档长度归一化参数

`b` 决定文档长度在词频归一化中的影响程度。[5][6][14]

- `b = 0`：不进行长度归一化；
- `b = 1`：按相对文档长度进行完全归一化；
- 常见经验设置为 `b = 0.75`。[3][5][6][14]

在长篇正文、短标题、产品名、日志、代码片段等字段之间，合适的 `b` 值可能差异很大。因此，默认 `b = 0.75` 不应被理解为普适最优值，而是广泛采用的经验初值。

关于中文语料、分词粒度、字段长度和参数选择的实际问题，可参见 [[queries/中文RAG中的BM25如何调优]]。

## 原始 BM25 与常见扩展

### 查询词频与相关反馈

标准展示公式通常只显式包含文档词频、文档长度和 IDF；不过 Stanford 课程材料指出，BM25 还可结合查询词项加权及（伪）相关反馈。[5]

这意味着“BM25”在文献中有时指核心文档项评分形式，有时也指包含查询加权、反馈或其他检索证据的完整概率排序方案。比较不同论文、工具或实验结果时，需明确是否使用了这些附加组件。

### 字段化与虚拟区域

Semantic Scholar 对 2012 年 SIGIR 研究的摘要提到 “virtual region” 概念：文档中的一部分可以像 BM25F 的字段一样提供相关性证据，并由可调权重控制其贡献；该研究报告这种做法对检索效果有益。[2]

这表明面向标题、正文、锚文本、元数据或结构化字段的检索，通常不应机械地把所有文本拼接后使用单一 BM25 评分。字段化变体可为不同区域赋予不同的匹配权重与长度统计，但本组来源没有提供 BM25F 的完整公式、原始论文书目信息或可复现实验设置，尚不足以据此判断其相对标准 BM25 的普遍收益。[2]

### 邻近度扩展

收录文献包括 Tao 与 Zhai 的 *“An exploration of proximity measures in information retrieval”*，以及 Song 等人将 BM25 扩展为利用词项邻近度的研究。[1][2] 这些工作反映出：标准 BM25 是以词项为中心的加性模型，并不天然表达查询词之间的距离、顺序或短语紧凑程度。

实际系统可将短语查询、邻近度分数或后续 [[重排序]] 与 BM25 结合，以弥补这一限制；但这些方法应视作 BM25 之上的扩展，而非标准核心公式的一部分。[2]

### BM25L

`BM25LSimilarity` 是一个以 BM25L 名义提供的实现变体。Atlassian Confluence 6.2.0 API 显示，其构造参数包括：

- `k1`：控制非线性词频归一化（饱和）；
- `b`：控制文档长度对词频归一化的影响；
- `d`：`shift parameter`。[9]

该实现默认设置为：

```text
k1 = 1.25
b = 0.4
d = 0.5
```

并使用与 Lucene `BM25Similarity` 相同的平滑 IDF 形式。[9]

现有来源仅把 `d` 描述为移位参数，没有给出完整 BM25L 公式、原始提出论文或其具体理论动机。因此，不能仅凭该 API 文档断言 BM25L 的全部计算细节，也不能将其默认参数解释为对所有语料优于标准 BM25 的推荐。[9]

## Lucene 的 BM25 实现

### `BM25Similarity`

[[Lucene]] 从至少 4.0.0 起提供 `BM25Similarity`。Lucene 4.0.0 API 明确说明：

- 默认 `k1 = 1.2`；
- 默认 `b = 0.75`；
- `k1` 控制非线性词频归一化/饱和；
- `b` 控制文档长度对词频值归一化的程度；
- IDF 为 `log(1 + (numDocs - docFreq + 0.5)/(docFreq + 0.5))`。[6]

Lucene 8.1.1 与 9.0.0 的 API 保持了相同的默认 `k1`、`b` 和 IDF 定义。[7][8] 因而，就这三代 API 文档所显示的公开评分参数而言，Lucene 的默认 BM25 配置具有较强的版本连续性。

### 平均字段长度

Lucene 8.1.1 和 9.0.0 中，`avgFieldLength(CollectionStatistics)` 的默认计算为：

```text
sumTotalTermFreq / docCount
```

即平均字段长度基于字段总词频与文档计数计算。[7][8] 这意味着 BM25 的长度归一化不是抽象地针对“整个 JSON 文档”进行，而是依赖具体字段的索引统计信息。字段分割方式、分析器的 token 化结果以及统计覆盖范围均会影响实际得分。

### `discountOverlaps`

Lucene 的 `discountOverlaps` 控制位置增量为零的 overlap token 是否从文档长度计算中排除。[6][7][8]

- 在 Lucene 4.0.0 和 8.1.1 的文档中，该字段说明为：位置增量为零的 token 可以在计算 norm 时被忽略。[6][7]
- Lucene 9.0.0 明确说明默认构造器的 `discountOverlaps = true`。[8]

该选项特别影响使用同义词、同位置多 token 分析结果的场景。若 overlap token 被计入长度，可能改变 [[文档长度归一化]]；若被忽略，则同义词扩展不会以相同方式拉长字段。因而，两个系统即便 `k1`、`b` 相同，只要分析器或 overlap 处理不同，也未必得到相同分数。

### 与文献公式的一致性边界

Stack Overflow 的回答认为 Lucene 是对文献 BM25 的实现，并提醒提问中的公式遗漏了 IDF 项。[10] 这一判断可作为实践性说明，但并非一手技术规范。

更严格地说，Lucene `BM25Similarity` 与常见 BM25 的词频饱和和长度归一化结构一致，但其具体评分还受以下实现选择影响：

- 平滑且非负的 IDF 形式；[6][7][8]
- 每字段统计；[7][8]
- norm 编码与解码机制；[6][9]
- `discountOverlaps`；[6][7][8]
- 查询解析、分析器、boost、短语匹配和其他搜索引擎评分组合机制。

因此，“Lucene BM25 等于论文公式”只在核心形式的层面近似成立；对逐分值复现或跨系统比较而言，应进一步检查版本、字段统计、索引配置与查询执行细节。

## Elasticsearch 中的 BM25

### 默认地位

Elasticsearch 官方文档说明，`BM25` 是 Elasticsearch 和 [[Lucene]] 默认使用的 Okapi BM25 算法；`similarity` 可按字段配置，`text` 与 `keyword` 等文本字段类型支持该配置。[13]

官方文档还将 BM25 描述为具备内置 TF 归一化、并被认为适用于短字段（例如名称）的 TF/IDF 型相似度。[14] 这一表述应理解为产品文档的工程性建议，而非对所有短字段任务均有优势的严格理论结论。

### 可配置参数

Elasticsearch 的 BM25 相似度设置包括：[14]

| 参数 | 默认值 | 作用 |
|---|---:|---|
| `k1` | `1.2` | 控制非线性词频归一化，即 [[词频饱和]] |
| `b` | `0.75` | 控制 [[文档长度归一化]] 程度 |
| `discount_overlaps` | `true` | 是否在计算 norm 时忽略位置增量为零的 token |

自定义相似度可在索引设置中定义，并再被字段映射引用。概念性示例如下：

```json
PUT /index
{
  "settings": {
    "index": {
      "similarity": {
        "my_bm25": {
          "type": "BM25",
          "k1": 0.9,
          "b": 0.0,
          "discount_overlaps": true
        }
      }
    }
  }
}
```

随后可在字段 mapping 中通过 `"similarity": "my_bm25"` 引用该配置。[14] Stack Overflow 中也给出了以 `type: BM25`、`b: 0`、`k1: 0.9` 定义自定义相似度的示例，但配置生命周期和是否需要重建索引的讨论属于社区回答，实施时应以对应 Elasticsearch 版本的官方文档为准。[12]

### 作为关键词检索器的使用

Haystack 的 `ElasticsearchBM25Retriever` 被描述为基于关键词的 Retriever：它在 `ElasticsearchDocumentStore` 中按照 BM25 计算两个字符串的加权词语重叠来获取文档。[11] 该文档特别指出，这种匹配适合人名、产品名、ID 和定义明确的错误信息等精确匹配需求；同时，BM25 轻量、简单，并且在分布外数据上有时不易被更复杂的 embedding 方法超越。[11]

这一实践定位说明，在 [[向量检索]] 与 [[RAG]] 系统中，BM25 不只是传统基线，也可作为处理稀有实体词、标识符、精确字符串和词法约束的重要召回通道。与 [[向量检索]] 结合时，可形成 [[混合检索]]，再通过 [[重排序]] 整合候选结果。

## 实现与实验中的关键差异

### 不应只比较 `k1` 与 `b`

BM25 部署中最常见的误区是仅比较 `k1` 与 `b`，忽略其他决定性因素。实际效果和分值还取决于：

1. **分析器与 token 化**：大小写、停用词、词干化、同义词、中文分词和 n-gram 会直接改变词频、文档频率及长度；
2. **字段边界**：标题与正文分开建字段，还是合并为一个字段，会改变平均长度与 IDF 统计；
3. **IDF 公式**：是否使用 `+0.5` 平滑、是否使用 `log(1 + ...)`；
4. **统计口径**：`docCount`、含字段文档数和总文档数是否一致；
5. **overlap token 处理**：`discount_overlaps` 的配置；
6. **查询构造**：布尔逻辑、短语查询、字段 boost、minimum_should_match、过滤条件与查询扩展；
7. **排序管线**：BM25 是否仅用于候选召回，之后是否经过 [[重排序]]。

故而，不能仅因两个系统都声称“使用 BM25”便预期其排序完全一致。

### 默认参数不是通用最优参数

`k1 = 1.2`、`b = 0.75` 同时出现在 Stanford 课程材料、Lucene API 及 Elasticsearch 文档中，是高度普及的默认组合。[5][6][7][8][14] 但这只能证明其是稳定、常见的工程默认值，不能证明对任意领域、语言、字段或评价指标都最优。

例如：

- 超短字段可能需要较弱甚至关闭的长度归一化；
- 长文档集合可能需要针对长度分布调整 `b`；
- 词项大量重复的领域文本可能需要重新测试 `k1`；
- 中文场景中，分词粒度本身会改变 $|D|$、`avgdl` 与 $f(q_i,D)$，从而影响参数含义。

参数调优应使用任务相关的验证集、明确的相关性标注或可靠的离线/在线指标，而不是仅复制默认值。中文 RAG 实践建议另见 [[queries/中文RAG中的BM25如何调优]]。
