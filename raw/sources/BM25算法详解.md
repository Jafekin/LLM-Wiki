# BM25：让搜索“更懂重点”的经典算法

> 当你在知识库里搜索“Redis 集群哨兵配置”，系统为什么能把最相关的文档排在前面？
>
> 在许多搜索引擎、文档检索系统和 RAG 应用的背后，BM25 往往就是那个可靠、快速且经典的基础排序器。

在做搜索、问答系统或 RAG（检索增强生成）时，我们经常会遇到一个问题：**用户输入了一句话，系统该从海量文档里挑出哪些内容？**

最朴素的做法是看关键词是否出现。但这显然不够：一篇文章里出现一次关键词，和出现十次关键词，相关性一样吗？一篇一万字的长文和一篇三百字的短文都出现了同样的关键词，应该排谁在前面？“的”“系统”这类高频词，真的该和“BM25”“Elasticsearch”这类专业词拥有相同权重吗？

BM25 就是为解决这些问题而生的。

---

## 一、BM25 是什么？

BM25 的全称是 **Best Matching 25**。它是一种经典的信息检索（Information Retrieval）排序算法，用来衡量：

> 一篇文档与用户查询到底有多相关？

系统会为每篇候选文档计算一个分数，再按分数从高到低排序。分数越高，意味着这篇文档越可能符合用户的搜索意图。

它常见于以下场景：

- 搜索引擎的关键词检索；
- 企业知识库搜索；
- 文档管理与全文检索；
- RAG 系统中的召回阶段；
- Elasticsearch、OpenSearch、Lucene 等检索引擎的文本排序。

BM25 并不理解语言的深层语义，但它非常擅长处理**精确的关键词匹配**。例如报错信息、接口名、配置项、产品名、专业术语等场景，通常都很适合使用 BM25。

---

## 二、一个直观的问题：关键词出现得越多，文档就越相关吗？

不一定。

假设用户搜索：

```text
Python 爬虫
```

有两篇候选文档：

- **文档 A**：一篇 300 字的文章，其中“Python”出现 3 次，“爬虫”出现 2 次；
- **文档 B**：一篇 10,000 字的技术合集，其中“Python”出现 3 次，“爬虫”也出现 2 次。

如果只看关键词出现次数，两篇文档似乎一样相关。但从实际体验看，文档 A 更可能专门讨论 Python 爬虫；而文档 B 可能只是顺带提到它。

再举一个极端例子：如果某篇文章把“Python 爬虫”重复写了 100 次，它当然不应该因为重复堆砌关键词就排到第一名。

BM25 正是针对这些现实问题提出了三个关键判断：

1. **关键词出现过，说明可能相关；**
2. **关键词出现更多次会加分，但加分会逐渐变少；**
3. **长文档要适当降权，避免它们凭借篇幅占便宜。**

此外，它还会考虑一个重要因素：**关键词是否稀有。**

---

## 三、稀有词为什么更重要？

假设你在一个包含 10 万篇文章的知识库里搜索“系统”。几乎每篇文章都可能有这个词，因此它几乎不能帮助我们区分文档。

但如果你搜索“BM25”或“Sentinel”，情况就不同了。这些词只会出现在少量特定文档中，因此它们更能指向真正相关的内容。

BM25 用 **IDF（Inverse Document Frequency，逆文档频率）** 来描述一个词的稀有程度。

- 一个词出现在越多文档中，IDF 越低；
- 一个词越少见，IDF 越高；
- IDF 越高，查询词对最终得分的贡献越大。

常用的 IDF 公式如下：

\[
\operatorname{IDF}(q_i) =
\ln\left(
\frac{N - n(q_i) + 0.5}{n(q_i) + 0.5} + 1
\right)
\]

其中：

- \(N\) 是文档总数；
- \(n(q_i)\) 是包含词 \(q_i\) 的文档数量。

你不必死记这个公式。只需要记住：

> 一个词越稀有，它越有能力把真正相关的文档从大量普通文档中筛出来。

---

## 四、BM25 的完整公式长什么样？

BM25 的常见公式如下：

\[
\operatorname{BM25}(D,Q) =
\sum_{q_i \in Q}
\operatorname{IDF}(q_i)
\cdot
\frac{f(q_i,D) \cdot (k_1+1)}
{f(q_i,D) + k_1 \cdot \left(1-b+b\cdot\frac{|D|}{\operatorname{avgdl}}\right)}
\]

看起来复杂，但拆开后其实只是在做三件事：**给稀有词更多权重、让词频增长逐渐饱和、对文档长度做校正。**

| 符号 | 含义 |
|---|---|
| \(D\) | 当前被评分的文档 |
| \(Q\) | 用户查询 |
| \(q_i\) | 查询中的某一个词 |
| \(f(q_i,D)\) | 该词在当前文档中出现的次数 |
| \(|D|\) | 当前文档的长度 |
| \(\operatorname{avgdl}\) | 文档集合的平均长度 |
| \(k_1\) | 控制词频影响的参数 |
| \(b\) | 控制文档长度归一化的参数 |

如果公式让你感到抽象，可以把它理解为下面这句话：

> 文档中出现了用户关心的稀有关键词，会大幅加分；同一个词多次出现仍会加分，但不会无限加分；文档过长会受到适度惩罚。

---

## 五、词频饱和：为什么重复 100 次不会得到 100 倍分数？

这是 BM25 最实用的设计之一。

在普通的词频统计中，某个词出现 10 次，似乎就应该比出现 1 次重要 10 倍。但真实世界不是这样：文章反复出现同一个词，可能只是作者写作习惯，也可能是人为做 SEO 关键词堆砌。

BM25 的做法是让词频收益逐渐饱和：

- 从 0 次变成 1 次，意义很大；
- 从 1 次变成 3 次，相关性会继续提高；
- 从 50 次变成 100 次，额外收益很有限。

这让 BM25 的排序更符合人的直觉：一篇正常讨论主题的文档，不会轻易被一篇疯狂重复关键词的文档击败。

控制这一行为的参数是 \(k_1\)。它通常取值在 **1.2 到 2.0** 之间，实践中常用 `1.2` 或 `1.5`。

---

## 六、长度归一化：为什么长文档不能天然占优？

长文档包含更多词，因此也更容易碰巧包含用户搜索的关键词。

例如，一篇 50 页的技术手册提到了一次“Redis Sentinel”，并不一定比一篇专门讲 Redis 高可用的短文更相关。如果不对长度做调整，长文档往往会因为拥有更多关键词出现机会而占据优势。

BM25 使用文档长度归一化来修正这一点：

- 文档比平均长度更长，得分会被适当压低；
- 文档较短且关键词集中，得分会更有优势；
- 参数 \(b\) 控制这一修正的强弱。

常见设置是：

```text
k1 = 1.2
b  = 0.75
```

其中：

- `b = 0`：完全不考虑文档长度；
- `b = 1`：充分按文档长度归一化；
- `b = 0.75`：是检索实践中常见的折中值。

---

## 七、BM25 和 TF-IDF 有什么区别？

很多人会把 BM25 看作 TF-IDF 的升级版本，这个理解大体没有问题。

TF-IDF 已经告诉我们两件重要的事：

1. 一个词在当前文档中出现得多，通常更重要；
2. 一个词在整个语料库中越少见，通常越有区分力。

而 BM25 在此基础上进一步改进：

| 对比维度 | TF-IDF | BM25 |
|---|---|---|
| 稀有词权重 | 使用 IDF | 使用 IDF |
| 词频影响 | 通常是线性或简单变换 | 采用词频饱和，更贴近实际 |
| 文档长度 | 依赖具体实现 | 内置长度归一化机制 |
| 工程应用 | 基础有效 | 更常用、更稳健的检索基线 |

一句话总结：

> TF-IDF 关注“词出现得多不多、词本身稀不稀有”；BM25 进一步考虑了“重复出现是否真的有价值”和“长文章是否占了篇幅便宜”。

---

## 八、BM25 在 RAG 中扮演什么角色？

在 RAG 系统中，大语言模型并不会直接记住企业内部的全部文档。通常的做法是：先根据用户问题从知识库中检索相关内容，再把这些内容放进模型上下文，让模型基于证据生成回答。

BM25 往往用于这个流程的前半段：

```text
用户提出问题
   ↓
将问题拆分或分词
   ↓
BM25 对知识库中的文档块评分
   ↓
取分数最高的 Top-K 文本块
   ↓
把文本块发送给大语言模型
   ↓
模型根据上下文生成回答
```

例如用户提问：

```text
如何配置 Redis 集群哨兵？
```

BM25 很擅长找到包含以下词语的文档片段：

```text
Redis、集群、Sentinel、哨兵、配置
```

尤其是对于以下内容，BM25 通常很有效：

- 错误码、报错堆栈；
- API 名称、函数名、类名；
- 配置键和命令行参数；
- 专有名词、产品名称和版本号；
- 用户直接复制粘贴的原始关键词。

---

## 九、BM25 的局限：它不真正理解“语义”

BM25 强在关键词，但它的弱点也来自关键词。

假设用户问：

```text
缓存高可用方案怎么做？
```

而知识库里的文档标题是：

```text
Redis Sentinel 部署指南
```

如果两者没有明显的重合词，BM25 就可能检索不到正确文档。因为它不知道“缓存高可用方案”和“Redis Sentinel”在语义上可能高度相关。

这也是为什么今天的 RAG 系统往往采用**混合检索**：

- **BM25**：负责关键词精确匹配；
- **向量检索**：负责语义相似匹配；
- **重排序模型（Reranker）**：对候选结果进行更精细的排序。

这种组合通常比只使用 BM25 或只使用向量检索更稳定。

---

## 十、结语

BM25 不是最新潮的算法，也不依赖复杂的深度学习模型，但它依然是搜索和 RAG 系统中的重要基础设施。

它的价值在于足够朴素，也足够实用：

- 稀有关键词更重要；
- 关键词重复不会无限加分；
- 长文档不会因为篇幅大而天然占优；
- 无需训练，速度快，解释性强。

如果你正在搭建一个知识库问答、企业搜索或 RAG 系统，可以把 BM25 视为一个非常可靠的起点。即使后续加入向量检索和重排序模型，BM25 也通常值得保留，作为混合检索体系中的关键词检索能力。

> 真正好的检索，不只是找到“包含这些词”的文档，而是尽可能找到“最可能解决用户问题”的内容。BM25 虽然简单，却为这件事提供了一个经典而坚实的起点。
